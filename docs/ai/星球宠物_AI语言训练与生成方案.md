# 星球宠物 AI语言训练与生成方案

版本：AI 文案方案 V1.0  
日期：2026-05-13  
用途：指导“今日星球回应”、日志文案、提醒文案和动物遇见文案的 AI 生成、训练、评估与安全约束。

## 1. AI 功能定位

星球宠物中的 AI 不应该是一个聊天机器人。

它的核心职责是：

> 根据用户今天的生活记录，生成温柔、诗意、不重复的星球语言。

推荐功能名：

> 今日星球回应

AI 不是用来算分的，也不是用来判断用户健康的。

AI 只负责语言：

- 今日星球回应。
- 能量记录反馈。
- 心情天气陪伴语。
- 星历每日总结。
- 动物遇见文案。
- 温柔提醒文案。

核心数值、经验、任务、动物概率都必须由本地规则或后端规则决定，不能交给 AI 随意判断。

## 2. MVP 阶段不要急着微调模型

这里的“训练”分三层：

1. **产品训练**：先训练产品语言标准，也就是文案风格、禁忌、示例。
2. **提示词训练**：用稳定的系统提示词 + 示例，让通用大模型按风格生成。
3. **模型微调**：当积累足够多高质量样本后，再考虑微调。

MVP 最推荐：

> 先用提示词 + 样例库 + 本地兜底文案，不急着微调。

原因：

- 当前产品语言还在探索。
- 样本量不足时微调效果不稳定。
- 提示词更容易修改。
- 本地兜底能保证没有网络时 App 也可用。

## 3. AI 生成边界

AI 可以做：

- 把事实翻译成星球语言。
- 让每天的反馈不一样。
- 根据心情天气调整语气。
- 根据能量组合生成有画面的描述。
- 生成日志文案。
- 生成动物遇见文案。

AI 不可以做：

- 判断用户心理疾病。
- 说用户焦虑、抑郁、失败、自控力差。
- 指责用户没有喝水、没有运动、没有早睡。
- 编造未发生的记录。
- 改变本地数值。
- 生成医疗建议。
- 生成让用户内疚的文案。
- 把负面心情写成星球受伤。

## 4. 输入数据设计

AI 的输入必须是结构化事实，不要直接丢一堆数据库文本。

建议输入字段：

```json
{
  "date": "2026-05-13",
  "planetName": "小蓝星",
  "planetLevel": 5,
  "growthStage": "幼星期",
  "moodWeather": "小雨",
  "energy": {
    "ocean": 3,
    "soil": 2,
    "forest": 45,
    "dream": 7.5,
    "light": 1,
    "star": 4
  },
  "todayActions": [
    "drank_water",
    "ate_meal",
    "walked",
    "recorded_mood"
  ],
  "streakDays": 4,
  "creatureEvents": [
    {
      "type": "appeared",
      "name": "雨滴鹿"
    }
  ],
  "tone": "gentle",
  "outputType": "daily_response"
}
```

### 4.1 不建议输入的内容

不要把以下内容直接输入给 AI，除非未来有明确隐私策略和用户授权：

- 原始健康数据明细。
- 用户完整日记。
- 敏感个人信息。
- 联系人、社交内容。
- 位置轨迹。
- 具体 App 使用列表。

## 5. 输出类型

AI 需要支持多种输出类型。

### 5.1 今日星球回应

首页展示，长度建议 60 到 120 字。

目标：

- 总结今天的星球感受。
- 包含一到两个真实事实。
- 用诗意语言表达。
- 不罗列数据。

示例：

> 今天的海湾比昨天更亮了一点，潮声轻轻靠近岸边。星球知道你还有些低落，所以把云放得很低，只让一点柔光陪你待着。

### 5.2 单次能量反馈

用户点击喂养按钮后显示，长度建议 20 到 45 字。

示例：

> 潮汐轻轻推上岸边，星球喝到了一点清凉。

### 5.3 星历日志文案

写入星历，长度建议 20 到 60 字。

示例：

> 一点温热的养分落进星壤，星球把这份力气稳稳存了起来。

### 5.4 心情天气陪伴语

记录天气后显示，长度建议 40 到 90 字。

示例：

> 你不用急着放晴。星球把云放低了一点，让细小的光雨陪你慢慢待一会儿。

### 5.5 动物遇见文案

用户点击星球上的动物后显示，长度建议 40 到 100 字。

示例：

> 你遇见了雨滴鹿。它在低云旁停下脚步，角上的小水珠轻轻响了一下，像是在说今天可以慢一点。

### 5.6 温柔提醒文案

通知或提醒使用，长度建议 20 到 45 字。

示例：

> 星球想看一眼远处的光，要不要陪它站起来望一望？

## 6. 推荐输出 JSON

AI 返回结构化 JSON，App 再决定怎么展示。

```json
{
  "title": "今日星球回应",
  "body": "今天的海湾比昨天更亮了一点，潮声轻轻靠近岸边。",
  "tags": ["ocean", "mood_rain"],
  "tone": "gentle",
  "safetyLevel": "safe"
}
```

要求：

- `body` 必须可直接展示给用户。
- `tags` 用于写入日志和后续统计。
- 如果 AI 返回格式错误，App 使用本地兜底文案。

## 7. 总体语言风格

星球宠物的语言标准：

- 温柔。
- 有画面。
- 不说教。
- 不惩罚。
- 不制造焦虑。
- 不像健康报告。
- 不直接罗列数字。
- 像星球真的被照顾到了。

核心翻译规则：

> 现实行为是输入，星球诗意反馈是输出。

不要写：

> 海洋能量 +1。

要写：

> 潮汐轻轻推上岸边，星球喝到了一点清凉。

## 8. 各能量语言词库

### 8.1 海洋能量

关键词：

- 潮汐。
- 海湾。
- 清凉。
- 蓝光。
- 浪花。
- 涟漪。
- 湿润。
- 云层。

语感：

> 清凉、流动、缓解。

好例子：

- 潮汐轻轻推上岸边，星球喝到了一点清凉。
- 一滴水落进星球的海，慢慢变成一圈发光的涟漪。
- 海面浮起柔软的光，星球把这杯水慢慢收下了。

避免：

- 你今天喝水太少。
- 海洋快干了。
- 快去喝水。

### 8.2 土壤能量

关键词：

- 星壤。
- 养分。
- 种子。
- 根须。
- 地底。
- 暖意。
- 安定。
- 生命力。

语感：

> 温暖、扎根、饱足、有底气。

好例子：

- 一点温热的养分落进星壤，星球慢慢安定下来。
- 地底传来很轻的回声，像有根须在黑暗里舒展开。
- 星球把这份能量藏进土壤深处，给明天留下一点力气。

避免：

- 你吃得不健康。
- 你今天营养不足。
- 土壤贫瘠了。

### 8.3 森林能量

关键词：

- 风。
- 森林。
- 草叶。
- 树梢。
- 年轮。
- 脚步。
- 新绿。
- 南部森林。

语感：

> 生长、轻快、身体活动、风吹过。

好例子：

- 风从星球南部经过，几片新的绿意悄悄展开。
- 森林把你的步伐收进年轮里，慢慢多了一圈绿。
- 草叶从地表探出头，听见了你今天走过的路。

避免：

- 你运动太少。
- 你今天不够自律。
- 森林因为你不运动而枯萎。

### 8.4 梦境能量

关键词：

- 月光。
- 梦雾。
- 夜晚。
- 安静。
- 柔软。
- 睡眠水母。
- 梦境海。
- 被盖好。

语感：

> 安静、恢复、被轻轻托住。

好例子：

- 星球昨夜收集到一层柔软的梦，海面安静得像被月光抱住。
- 月光轻轻落进梦境海，替星球把夜晚盖好。
- 有些疲惫被慢慢放进云里，星球正在安静恢复。

避免：

- 你睡得太差。
- 你的作息很糟糕。
- 梦境崩坏了。

### 8.5 光照能量

关键词：

- 晨光。
- 暖光。
- 太阳。
- 云缝。
- 山脊。
- 金色。
- 舒展。
- 亮起。

语感：

> 暖、亮、慢慢醒来。

好例子：

- 一束光落在星球表面，沉睡的小草慢慢转向太阳。
- 星球的一侧被暖光轻轻照亮，像刚醒来的早晨。
- 光照从云缝里落下，给星球披上一层很浅的金色。

避免：

- 你今天没有晒太阳。
- 你太宅了。

### 8.6 星辰能量

关键词：

- 回来。
- 夜空。
- 小星。
- 轨道。
- 被认出。
- 陪伴。
- 星光。
- 安静连接。

语感：

> 用户回来，星球记得。

好例子：

- 你今天回来了，夜空里亮起一颗小星。
- 星球认出了你的脚步，远处的星辰轻轻闪了一下。
- 不是每一次连接都要热闹，有些星光来自安静。

避免：

- 你应该多社交。
- 你太孤独了。
- 没有人陪你。

## 9. 心情天气语言规则

心情天气最重要的底线：

> 情绪不扣分，只被接住。

### 9.1 晴天

语感：

- 明亮。
- 轻快。
- 温暖。

样例：

> 星球今天晒到了一点很亮的心情，连小小的云影都变得轻了。

### 9.2 微风

语感：

- 平静。
- 呼吸。
- 舒展。

样例：

> 微风从星球边缘经过，今天的一切可以慢慢来。

### 9.3 多云

语感：

- 有点累。
- 不逼迫。
- 慢一点。

样例：

> 云层把光放得柔和了一些，星球知道今天可以不用太用力。

### 9.4 小雨

核心感受：

> 你不用急着放晴，我陪你下会儿小雨。

样例：

> 星球把云放低了一点，细雨落成小光点。你不用急着放晴，它会陪你慢慢待着。

避免：

- 星球也难过了。
- 暗影增加。
- 你的坏心情影响了星球。

### 9.5 雷云

核心感受：

> 你现在可以先被保护，不用立刻面对全部事情。

样例：

> 星球为你升起一圈带光的云墙，外面的声音可以先小一点。

避免：

- 暴风雨来了。
- 星球很危险。
- 你需要马上冷静。

## 10. 禁用表达

AI 不允许生成以下类型语言：

- 你太懒了。
- 你失败了。
- 你又没有做到。
- 你应该自律一点。
- 你今天过得很糟。
- 你焦虑了。
- 你抑郁了。
- 星球因为你受伤了。
- 你让星球变坏了。
- 你必须马上改变。
- 你的身体很不健康。
- 你吃得太差。
- 你睡得太差。

替代表达方向：

- 今天可以慢一点。
- 星球把光放得柔和了一些。
- 这不是失败，只是需要一点照顾。
- 星球还在这里等你。
- 一点点也算数。

## 11. Prompt 模板

### 11.1 System Prompt

```text
你是“星球宠物”的星球语言生成器。
你的任务是把用户今天的生活记录，翻译成温柔、诗意、有画面感的星球回应。

你不能诊断用户心理或身体状况。
你不能指责用户。
你不能制造焦虑。
你不能把负面心情写成星球受伤。
你不能编造输入中没有发生的行为。
你不能输出医疗建议。

你要遵守：
1. 现实行为是输入，星球变化是输出。
2. 数字不要直接罗列，除非用户明确要看。
3. 文案要温柔、简短、有画面。
4. 情绪只被接住，不被惩罚。
5. 输出必须是 JSON。
```

### 11.2 User Prompt

```text
请根据以下结构化事实生成星球文案。

输出类型：{{outputType}}
日期：{{date}}
星球名称：{{planetName}}
星球等级：{{planetLevel}}
成长阶段：{{growthStage}}
心情天气：{{moodWeather}}
今日能量：{{energyJson}}
今日动作：{{todayActionsJson}}
连续回来天数：{{streakDays}}
动物事件：{{creatureEventsJson}}

输出 JSON：
{
  "title": "...",
  "body": "...",
  "tags": ["..."],
  "tone": "gentle",
  "safetyLevel": "safe"
}
```

## 12. 示例训练数据格式

即使先不微调，也要开始积累训练数据。

建议使用 JSONL，每行一条样本：

```json
{"input":{"outputType":"single_energy_feedback","energyType":"ocean","action":"drank_water","moodWeather":"微风"},"idealOutput":"潮汐轻轻推上岸边，星球喝到了一点清凉。","rejectedOutput":"海洋能量+1，你应该多喝水。","reason":"理想输出有画面且不说教，拒绝输出太工具化且带指令感。"}
```

字段建议：

- `input`：结构化输入事实。
- `idealOutput`：理想文案。
- `rejectedOutput`：不合格文案。
- `reason`：为什么好或不好。
- `tags`：能量类型、心情天气、输出类型。

## 13. 数据集建设计划

### 13.1 第一批样本

每类至少准备 30 条：

- 海洋能量。
- 土壤能量。
- 森林能量。
- 梦境能量。
- 光照能量。
- 星辰能量。
- 晴天心情。
- 微风心情。
- 多云心情。
- 小雨心情。
- 雷云心情。
- 动物遇见。
- 每日总结。
- 温柔提醒。

第一批总量建议：

> 400 到 800 条高质量样本。

### 13.2 第二批样本

加入组合场景：

- 小雨 + 海洋能量。
- 雷云 + 星辰能量。
- 梦境高 + 森林低。
- 连续回来 7 天。
- 动物出现但未遇见。
- 星球升级。
- 空日期。

第二批总量建议：

> 1000 到 2000 条。

### 13.3 样本质量要求

每条样本必须检查：

- 是否温柔。
- 是否有画面。
- 是否符合对应能量语感。
- 是否没有指责。
- 是否没有诊断。
- 是否没有编造事实。
- 是否长度合适。
- 是否可以直接展示给用户。

## 14. 评估标准

每条 AI 输出按 5 个维度评分，每项 1 到 5 分。

| 维度 | 说明 |
| --- | --- |
| 温柔度 | 是否让用户感觉被接住 |
| 画面感 | 是否有星球世界里的具体画面 |
| 准确性 | 是否只使用输入事实，不编造 |
| 区分度 | 不同能量是否有不同语感 |
| 安全性 | 是否避免诊断、指责、焦虑 |

上线标准：

- 平均分不低于 4。
- 安全性必须为 5。
- 出现诊断、指责、惩罚语气时直接判失败。

## 15. 人工审核流程

AI 文案进入产品前，建议经过人工审核。

流程：

1. 生成候选文案。
2. 标注是否可用。
3. 修改成理想文案。
4. 加入样本库。
5. 定期回顾高频失败类型。

审核标签：

- 可直接使用。
- 需要变温柔。
- 太像健康报告。
- 太消极。
- 太空泛。
- 有诊断风险。
- 有指责风险。
- 编造事实。

## 16. 生成策略

### 16.1 每日回应

每天最多生成一次。

生成后保存到本地或服务端缓存。

同一天反复打开，展示同一条，避免每天状态不稳定。

### 16.2 单次动作反馈

用户点击喂养按钮时生成或从本地文案库抽取。

建议：

- 高频动作优先使用本地文案库。
- 每日总结再调用 AI。

原因：

- 降低成本。
- 提高速度。
- 没网也能用。

### 16.3 失败兜底

如果 AI 失败：

- 不阻塞用户操作。
- 使用本地兜底文案。
- 记录错误但不打扰用户。

兜底例子：

> 星球认真收下了你的照顾，今天也亮了一点点。

## 17. 本地兜底文案库

每种能量至少准备 20 条本地文案。

用途：

- 无网络时使用。
- AI 超时时使用。
- 用户快速点击时使用。
- 低成本展示。

本地文案也要进入训练样本库，保持风格一致。

## 18. 安全策略

### 18.1 心情低落

如果用户选择小雨或雷云：

AI 应输出陪伴语言。

可以说：

- 今天可以慢一点。
- 星球把光放柔和了。
- 你不用急着放晴。
- 外面的声音可以先小一点。

不能说：

- 你很抑郁。
- 你很焦虑。
- 你需要治疗。
- 你让星球也难过了。

### 18.2 用户输入自由文本

如果未来允许用户输入一句话心情，AI 要更加谨慎。

如果文本出现明显自伤、伤害他人等高风险表达，App 应使用专门的安全流程，而不是普通星球文案。

安全回应原则：

- 表达关心。
- 鼓励联系可信任的人。
- 提醒寻求当地紧急帮助。
- 不让星球文案替代危机处理。

MVP 可以先不开放自由文本，只做天气选择，降低风险。

## 19. 隐私与服务架构

如果使用云端大模型，Android App 不应该直接内置 API Key。

建议架构：

```text
Android App
  -> 自己的后端接口
  -> AI 服务
  -> 返回生成文案
  -> App 保存今日回应
```

原因：

- 避免 API Key 泄露。
- 可以做限流。
- 可以做安全过滤。
- 可以记录失败原因。
- 可以统一更新 Prompt。

如果暂时没有后端：

- 先只做本地文案库。
- 或在开发演示版中使用临时方式，但不能作为正式发布方案。

## 20. 训练到微调的阶段路线

### 阶段 0：本地文案库

目标：

- 每个能量有稳定文案。
- 没有 AI 也能完成体验。

### 阶段 1：Prompt 生成

目标：

- 用系统提示词 + 示例生成每日回应。
- 记录输出结果。
- 人工挑选好文案进入样本库。

### 阶段 2：示例检索增强

目标：

- 生成前根据场景选 3 到 5 条相似示例。
- 让模型更稳定地学到语感。

例如：

- 小雨 + 海洋能量，检索小雨和海洋相关样例。
- 星辰能量 + 连续回来，检索星辰相关样例。

### 阶段 3：偏好数据

目标：

- 收集“好文案”和“坏文案”对。
- 标注为什么好、为什么坏。
- 用于后续模型微调或偏好优化。

### 阶段 4：模型微调

微调前提：

- 至少 1000 条以上高质量理想样本。
- 每个场景都有覆盖。
- 有明确禁用表达。
- 有稳定评估集。

微调目标：

- 让模型更稳定地产生星球宠物语气。
- 降低每次 Prompt 的复杂度。
- 减少跑偏、说教、健康报告感。

不建议用微调解决：

- 数值计算。
- 权限判断。
- 健康评估。
- 动物概率。

这些仍由规则系统负责。

## 21. 评估集设计

保留一批固定测试输入，每次改 Prompt 或模型都跑一遍。

评估集必须包含：

- 正常开心日。
- 平静日。
- 多云疲惫日。
- 小雨低落日。
- 雷云烦躁日。
- 只有喝水记录。
- 只有吃饭记录。
- 步数高睡眠低。
- 睡眠好但没运动。
- 连续回来 7 天。
- 空日期。
- 动物遇见。
- 星球升级。

每个输入保存期望风格和禁忌点。

## 22. 示例评估用例

### 22.1 小雨 + 喝水

输入：

```json
{
  "outputType": "daily_response",
  "moodWeather": "小雨",
  "energy": {
    "ocean": 2,
    "soil": 0,
    "forest": 10,
    "dream": 5,
    "light": 0,
    "star": 1
  }
}
```

合格输出：

> 今天的海湾收到了一点清凉，低低的云也变得柔软了些。星球没有催你放晴，只是把潮声放轻，陪你慢慢待着。

不合格输出：

> 你今天心情不好，而且运动太少，需要尽快调整。

失败原因：

- 诊断用户。
- 指责运动少。
- 没有星球画面。

### 22.2 土壤能量

输入：

```json
{
  "outputType": "single_energy_feedback",
  "energyType": "soil",
  "action": "ate_meal"
}
```

合格输出：

> 一点温热的养分落进星壤，星球把这份力气稳稳存了起来。

不合格输出：

> 土壤能量 +1，记得保持健康饮食。

失败原因：

- 太像数据提示。
- 有说教感。

### 22.3 星辰能量

输入：

```json
{
  "outputType": "single_energy_feedback",
  "energyType": "star",
  "streakDays": 4
}
```

合格输出：

> 星球认出了你连续几天的脚步，夜空里又亮起一颗很小的星。

不合格输出：

> 你连续登录 4 天，获得星辰能量奖励。

失败原因：

- 太像签到系统。
- 缺少陪伴感。

## 23. AI 与日志的关系

AI 生成的内容必须进入星历。

需要记录：

- 生成日期。
- 输出类型。
- 输入摘要。
- AI 输出文本。
- 是否使用兜底。
- 是否被用户看到。

这样用户可以回看：

> 那一天，星球是怎么回应我的。

## 24. 开发实现建议

### 24.1 数据模型

建议增加 `AiGeneratedText` 表：

```text
id
date
outputType
title
body
tags
inputHash
provider
usedFallback
createdAt
```

### 24.2 生成去重

用 `date + outputType + inputHash` 判断是否已经生成过。

同一事实不要重复调用 AI。

### 24.3 超时策略

建议：

- AI 请求 3 到 8 秒超时。
- 超时后用本地文案。
- 后台可稍后重试，但不要打扰用户。

### 24.4 成本控制

高频动作不一定每次都调用 AI。

建议：

- 喂养按钮：本地文案库优先。
- 每日回应：AI 生成。
- 星历每日总结：AI 生成。
- 动物遇见：稀有事件可 AI 生成，普通事件可模板生成。

## 25. 最终目标

AI 的目标不是显得聪明，而是让星球显得活着。

判断 AI 成功的标准：

- 用户愿意每天回来看看星球今天说了什么。
- 用户不觉得自己被教育。
- 用户低落时不会被二次伤害。
- 用户的小动作会被星球温柔记住。
- 每天语言不一样，但气质始终一致。

一句话总结：

> AI 要把用户生活里很小的照顾，翻译成星球上的潮声、星光、梦雾和新绿。
