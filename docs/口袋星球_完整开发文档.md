# 口袋星球 APP 完整开发文档

版本：MVP V1.0  
目标平台：Android  
开发方式：原生 Android  
文档用途：用于指导 Gemini 分阶段生成代码、用于人工验收每一阶段成果、用于记录每次开发流程。

## 1. 项目定位

口袋星球是一款现实行为驱动的轻养成 Android App。用户的步行、专注、久坐、夜间活跃、规律作息、外出通勤等现实行为，会被转换为星球上的森林、水晶、梦境、城市、荒漠、暗影等生态变化。

产品第一版不是严肃医疗健康软件，也不是重度游戏。MVP 的目标是验证一个核心体验：用户不是机械打卡，而是在用自己的日常行为影响一颗小星球。

## 2. MVP 边界

### 2.1 必须实现

- 星球创建和本地保存。
- 星球首页展示。
- 六类生态值：森林值、水晶值、梦境值、城市值、荒漠值、暗影值。
- 手动记录步行、久坐、夜间活跃、通勤或外出。
- 专注计时。
- 行为转生态值。
- 星球状态判断。
- 星球事件日志。
- 基础图鉴解锁。
- 基础生态任务。
- 我的/设置页面。
- 本地数据持久化。
- 数据清空。

### 2.2 MVP 暂不实现

- 服务器。
- 登录注册。
- 云同步。
- 好友、排行、社区。
- AI 星球日报。
- Google Play 内购。
- 动态壁纸。
- 复杂 3D 星球。
- 强制 Activity Recognition。
- 强制定位。

### 2.3 技术选型

- 语言：Kotlin。
- UI：Jetpack Compose。
- 页面导航：Navigation Compose。
- 架构：MVVM。
- 本地数据库：Room。
- 设置存储：DataStore，若实现复杂可先用 SharedPreferences。
- 异步：Kotlin Coroutine + Flow。
- 通知：Android 本地通知，MVP 可后置到第 7 阶段。
- 视觉：MVP 使用 Compose Canvas、渐变圆形、图片资源或简单图层表现星球，不追求 3D。

## 3. 用户核心流程

### 3.1 首次使用流程

1. 用户打开 App。
2. 系统判断本地是否已有星球。
3. 如果没有星球，进入创建星球页。
4. 用户输入星球名称、昵称，选择初始风格。
5. 系统创建 Planet、UserSettings、默认图鉴和默认任务数据。
6. 用户进入星球首页。

验收标准：

- 首次打开必须进入创建流程。
- 创建完成后进入首页。
- 关闭并重新打开 App 后，不再重复创建。
- 首页可以显示已保存的星球名称和昵称。

### 3.2 日常使用流程

1. 用户进入首页查看星球状态。
2. 用户进入专注页完成一次专注，或在首页/记录页手动添加行为。
3. 系统更新 DailyStats。
4. 系统按照规则更新 Planet 生态值。
5. 系统生成 PlanetEvent 日志。
6. 系统检查任务进度。
7. 系统检查图鉴解锁条件。
8. 首页刷新星球状态、事件和建议行动。

验收标准：

- 每一种行为记录后，生态值有明确变化。
- 生态值始终保持在 0 到 100。
- 行为触发后能生成日志。
- 首页能看到最新变化。

## 4. 信息架构与页面设计

界面实现必须结合 `docs/界面参考_UI实现规范.md` 和 `docs/assets/ui-reference/` 中的参考图。MVP 不要求完全复刻像素素材，但必须保持浅蓝天空背景、像素风气质、奶油色面板、星球生态主视觉和五栏底部导航。

### 4.1 底部导航

MVP 使用 5 个主 Tab：

1. 星球：首页，展示星球和生态状态。
2. 专注：计时、完成奖励、水晶事件。
3. 日志：按时间线展示星球事件。
4. 任务：每日生态任务和进度。
5. 我的：用户信息、图鉴入口、设置入口。

说明：参考图底部导航固定为星球、日志、专注、任务、我的。图鉴页面作为“我的”页面中的入口或二级页面，不放入第一版底部导航。

### 4.2 创建星球页

页面目标：让用户在 3 步内完成初始化。

页面元素：

- App 名称：口袋星球。
- 一句概念文案：你的生活，正在形成一颗星球。
- 星球名称输入框。
- 用户昵称输入框。
- 初始风格选择：默认、森林、梦境。
- 创建按钮。

交互规则：

- 星球名称不能为空。
- 昵称为空时默认使用“星际旅人”。
- 点击创建后保存数据并进入首页。

### 4.3 星球首页

页面目标：让用户第一眼看到“我的星球今天怎么样”。

页面元素：

- 顶部资料牌：App 名称、星球头像、星球名称、等级。
- 中部大型星球视觉区域。
- 生态摘要：森林、水晶、梦境、城市、荒漠、暗影。
- 今日主要事件：最多显示 3 条。
- 建议行动：根据短板显示一句建议。
- 手动记录入口：步行、久坐、夜间活跃、通勤。

状态文案示例：

- 生态繁荣：星球正在发出稳定的微光。
- 生态稳定：森林与水晶保持着温和共鸣。
- 轻度失衡：星球边缘出现了轻微波动。
- 生态恶化：荒漠和暗影正在扩大。
- 星球危机：生态系统需要立刻恢复。

### 4.4 专注页

页面目标：完成专注并把专注转化为水晶生态。

页面元素：

- 标题：专注模式。
- 时间选择：25 分钟、45 分钟、自定义。
- 倒计时圆环。
- 开始、暂停、继续、结束按钮。
- 完成提示：水晶塔完成一次充能。

业务规则：

- 用户开始后进入倒计时。
- 暂停时保留剩余时间。
- 提前结束时记录未完成 FocusSession，不给或少给奖励。
- 倒计时完成后增加水晶值，生成日志，更新任务和图鉴。

### 4.5 日志页

页面目标：让用户看到行为产生的世界观反馈。

页面元素：

- 日期选择。
- 时间线列表。
- 每条日志包含时间、事件类型、标题、描述、稀有度。

日志文案要求：

- 不直接写“你增加了 10 点森林值”作为主文案。
- 优先使用世界观表达，例如“南部森林的种子开始苏醒”。
- 数据可以作为副信息展示。

### 4.6 任务页

页面目标：把普通目标包装成星球任务。

默认任务：

- 让南部森林下雨：今日步行 30 分钟。
- 点亮水晶塔：完成 60 分钟专注。
- 阻止沙漠扩张：每 60 分钟起身活动一次。
- 召回睡眠水母：在理想休息时间前进入休息状态。

页面元素：

- 任务名称。
- 任务描述。
- 进度条。
- 奖励说明。
- 领取按钮或已完成状态。

### 4.7 图鉴页

页面目标：提供收集感和长期目标。

入口：从“我的”页面进入，也可以在后续版本独立成主 Tab。

初始图鉴：

- 森林鹿：步行累计 30 分钟解锁。
- 水晶猫：专注累计 60 分钟解锁。
- 睡眠水母：梦境值达到 60 解锁。
- 沙漠机械虫：荒漠值达到 50 解锁。
- 夜行蘑菇人：暗影值达到 50 解锁。

页面元素：

- 分类：生物、地貌、事件。
- 网格卡片。
- 已解锁展示名称、描述、稀有度、解锁时间。
- 未解锁展示剪影和条件。

### 4.8 我的/设置页

页面目标：管理用户资料、目标、主题和隐私。

页面元素：

- 用户昵称。
- 星球名称。
- 星球等级。
- 累计使用天数。
- 累计专注时间。
- 累计步行时间。
- 主题模式：白天、夜间、跟随系统。
- 每日步行目标。
- 每日专注目标。
- 久坐提醒间隔。
- 隐私说明。
- 清空本地数据。

## 6. 核心规则

### 6.1 数值边界

所有生态值更新后必须执行限制：

```kotlin
value.coerceIn(0, 100)
```

禁止出现负数，禁止超过 100。

### 6.2 行为转生态值

| 行为 | 计算规则 | 影响 |
| --- | --- | --- |
| 步行 | 森林值增加 walkingMinutes * 0.8 | 增加森林，少量降低荒漠 |
| 专注 | 水晶值增加 focusMinutes * 1.0 | 增加水晶，触发水晶事件 |
| 久坐 | 荒漠值增加 sedentaryHours * 8 | 增加荒漠 |
| 夜间活跃 | 暗影值增加 nightActiveMinutes * 1.2 | 增加暗影，降低梦境 |
| 按时休息 | 梦境值增加 15 | 增加梦境，降低暗影 |
| 通勤/外出 | 城市值增加 commuteMinutes * 0.4 | 增加城市 |

### 6.3 星球状态判断

```text
positive = forestValue + crystalValue + dreamValue
negative = desertValue + shadowValue
balanceScore = positive - negative

if balanceScore > 80:
    state = "生态繁荣"
elif balanceScore > 30:
    state = "生态稳定"
elif balanceScore >= -30:
    state = "轻度失衡"
elif balanceScore >= -80:
    state = "生态恶化"
else:
    state = "星球危机"
```

### 6.4 事件生成规则

- 步行分钟数达到 30：生成“森林降雨”。
- 专注分钟数达到 60：生成“水晶共鸣”。
- 久坐分钟数达到 120：生成“沙漠裂缝”。
- 夜间活跃分钟数达到 60：生成“暗影孢子”。
- 梦境值达到 60：生成“梦境潮汐”。
- 城市值达到 50：生成“港口点亮”。

事件必须写入日志页，并在首页今日事件中展示。

### 6.5 建议行动规则

- 如果 desertValue 最高：建议“起来活动几分钟，阻止荒漠继续扩张。”
- 如果 shadowValue 最高：建议“今晚早点休息，梦境海洋正在等待恢复。”
- 如果 crystalValue 低于 30：建议“完成一次专注，为水晶塔充能。”
- 如果 forestValue 低于 30：建议“记录一次步行，让南部森林重新生长。”
- 如果整体状态为生态繁荣：建议“星球状态很好，保持今天的节奏。”

## 7. 项目代码结构建议

```text
app/
  src/main/java/com/example/planetlife/
    MainActivity.kt
    PlanetLifeApp.kt
    navigation/
      AppNavGraph.kt
      AppRoute.kt
    data/
      local/
        AppDatabase.kt
        dao/
        entity/
      repository/
        PlanetRepository.kt
        FocusRepository.kt
        TaskRepository.kt
        CollectionRepository.kt
      settings/
        SettingsDataStore.kt
    domain/
      model/
      rules/
        EcologyRuleEngine.kt
        PlanetStateCalculator.kt
        EventGenerator.kt
        CollectionUnlocker.kt
        TaskProgressUpdater.kt
    ui/
      theme/
      onboarding/
      home/
      focus/
      logs/
      tasks/
      collection/
      profile/
      components/
```

原则：

- Activity 只负责承载 Compose。
- ViewModel 负责页面状态和用户动作。
- Repository 负责数据读写。
- RuleEngine 负责生态算法。
- UI 不直接写数据库逻辑。
- 算法不要散落在多个页面中。

## 8. 分阶段开发计划

### 阶段 0：项目初始化

阶段目标：创建可运行的 Android 项目骨架。

开发内容：

1. 创建 Kotlin + Compose Android 项目。
2. 配置 Gradle、Compose、Navigation、Room、DataStore。
3. 创建包结构。
4. 创建 `MainActivity` 和 `PlanetLifeApp`。
5. 创建基础主题。
6. 创建空白导航页面。
7. 参考 `docs/assets/ui-reference/` 建立浅蓝背景、奶油色面板、深蓝标题、绿色选中态等基础 UI token。

交付物：

- App 可以编译运行。
- 底部导航可以切换页面。
- 每个页面有占位 UI。

验收命令：

```text
./gradlew assembleDebug
```

阶段总结：

- 本阶段只证明项目能跑，不追求功能完整。
- 如果构建失败，优先修 Gradle 和依赖版本。

### 阶段 1：星球创建与首页静态 UI

阶段目标：完成首次创建星球流程和首页基础展示。

开发内容：

1. 判断是否已有星球。
2. 无星球时进入创建页面。
3. 输入星球名称、昵称。
4. 保存 Planet 和 UserSettings。
5. 首页展示星球名称、等级、状态、六类生态值。
6. 使用 Compose Canvas 或图片实现一个简单星球视觉。
7. 首页布局参考 `home.png`：顶部资料牌、中央星球、生态值卡片、今日事件、底部导航。

交付物：

- 首次打开进入创建页。
- 创建后进入首页。
- 重启后读取星球信息。

验收标准：

- 星球名称不能为空。
- 昵称可为空，默认“星际旅人”。
- 首页不能是纯数据表，必须有星球视觉区域。

### 阶段 2：Room 数据库与核心生态规则

阶段目标：建立本地数据层和行为转生态闭环。

开发内容：

1. 创建 Room Entity、Dao、Database。
2. 创建 Repository。
3. 创建 `EcologyRuleEngine`。
4. 实现手动记录行为：步行、久坐、夜间活跃、通勤。
5. 行为更新 DailyStats。
6. 行为更新 Planet 生态值。
7. 限制所有生态值在 0 到 100。

交付物：

- 用户可手动添加行为分钟数。
- 首页生态值实时变化。
- 数据重启后不丢失。

验收标准：

- 步行增加森林值。
- 久坐增加荒漠值。
- 夜间活跃增加暗影值并降低梦境值。
- 通勤增加城市值。
- 任何值不得小于 0 或大于 100。

### 阶段 3：星球日志系统

阶段目标：每次关键行为都能生成世界观日志。

开发内容：

1. 创建 PlanetEvent 表和 Dao。
2. 创建 `EventGenerator`。
3. 触发行为后生成日志。
4. 日志页按日期展示事件。
5. 首页展示今日最新 1 到 3 条事件。

交付物：

- 日志页有时间线列表。
- 行为完成后能看到新日志。
- 重启 App 后日志仍存在。

验收标准：

- 日志包含时间、标题、描述、事件类型。
- 文案不能只是数字变化。
- 至少实现森林、水晶、荒漠、暗影四类事件。

### 阶段 4：专注计时

阶段目标：完成专注计时、完成奖励和专注记录。

开发内容：

1. 专注页支持 25 分钟、45 分钟、自定义。
2. 实现开始、暂停、继续、结束。
3. 完成后写入 FocusSession。
4. 完成后增加水晶值。
5. 完成后生成水晶事件。
6. 提前结束记录为未完成。
7. UI 参考 `focus.png`：标题牌、大倒计时圆环、水晶塔视觉、开始/暂停/结束按钮、专注反馈卡片。

交付物：

- 专注倒计时可正常运行。
- 完成专注后水晶值变化。
- 日志页出现专注事件。

验收标准：

- 后台切换后计时状态尽量不丢失。
- MVP 若暂时无法做后台可靠计时，必须在文档或代码 TODO 中说明。

### 阶段 5：任务系统

阶段目标：每日任务推动用户完成核心行为。

开发内容：

1. 创建 PlanetTask 表。
2. 每日生成 2 到 4 个任务。
3. 行为记录后更新任务进度。
4. 任务完成后允许领取奖励。
5. 奖励影响生态值并生成日志。
6. UI 参考 `tasks.png`：大任务卡片、状态标签、进度条、奖励行。

交付物：

- 任务页展示任务列表。
- 进度条能变化。
- 奖励能领取。

验收标准：

- 任务名称必须使用星球世界观。
- 任务完成状态重启后不丢失。
- 领取奖励不能重复领取。

### 阶段 6：图鉴系统

阶段目标：根据行为和生态值解锁图鉴。

开发内容：

1. 创建 Creature 表。
2. 初始化默认图鉴。
3. 创建 `CollectionUnlocker`。
4. 行为后检查解锁条件。
5. 图鉴页展示已解锁和未解锁状态。
6. 新解锁时生成日志或 Toast/Snackbar。
7. UI 参考 `collection.png`：分类切换、两列网格、已解锁卡片、未解锁剪影卡片。

交付物：

- 图鉴页有网格。

验收标准：

- 未解锁项不能展示完整描述。
- 解锁后保存解锁时间。
- 同一图鉴不能重复解锁。

### 阶段 7：我的、设置、主题与隐私

交付物：

- 设置项可以保存。
- 重启后设置仍有效。
- 清空数据后回到创建星球流程。

## 9. 每阶段通用开发流程

每一轮让 Gemini 开发时都按下面流程执行：

1. 明确本轮阶段目标。
2. 限定本轮只改哪些模块。
3. 让 Gemini 先说明文件结构和实现方案。
4. 让 Gemini 生成或修改代码。
5. 运行构建。
6. 人工打开 App 验收核心流程。
7. 记录本轮完成项、问题、下一轮任务。
8. 把代码交给 Codex 复查。

每一轮必须输出：

- 本轮改了哪些文件。
- 实现了哪些功能。
- 没有实现哪些功能。
- 运行了什么验证命令。
- 当前已知问题。
- 下一阶段建议。

## 10. Gemini 开发约束

对 Gemini 的统一要求：

- 不要一次性做完所有功能。
- 不要引入服务器。
- 不要引入登录注册。
- 不要引入复杂 3D。
- 不要把业务逻辑写死在 Composable 里。
- 不要让 UI 直接操作 Dao。
- 不要跳过本地存储。
- 每次输出都要说明新增和修改的文件。
- 每次都要保证 `assembleDebug` 可以通过。
